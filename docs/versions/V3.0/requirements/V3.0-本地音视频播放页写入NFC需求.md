# 本地音视频播放页写入 NFC 需求文档
**所属产品**：NFC卡片助手  
**所属版本**：V3.0  
**文档版本**：V1.4  
**创建日期**：2026-04-23  
**最后更新**：2026-04-29  
**文档阶段**：开发中  
**功能优先级**：P0

> 说明：本文档只描述 V3.0 的产品需求、页面规则与验收标准。数据库、云函数、云部署和目录结构见技术方案文档。  
> 展示说明：播放页属于浏览器网页，不属于小程序页面栈；原型和 UI 需要按“小程序端 / 网页端”分开展示。

---

## 一、需求背景
V3.0 将原来的“本地音频直链写入 NFC”升级为“本地音视频播放页写入 NFC”。

目标是：
- 用户可以上传本地音频或视频文件
- 用户可以选填播放页标题
- 用户可以选择播放页风格
- NFC 写入的是固定播放页链接，而不是文件直链
- 浏览器打开后展示播放页并手动播放
- 删除文件后，旧链接打开时提示“链接不存在或已删除”

---

## 二、产品目标
### 2.1 核心目标
- NFC 打开的是播放页，而不是媒体文件直链
- 同一个文件可以再次生成分享链接并重新写入
- 音频和视频共用一套上传、管理和写卡流程
- 通过较小的文件限制控制成本
- 删除文件时采用真删除，不保留软删除数据

### 2.2 首版范围
| 能力 | 首版要求 |
|------|----------|
| 音频格式 | `mp3 / m4a / wav` |
| 视频格式 | `mp4` |
| 音频大小 | 不超过 `10MB` |
| 视频大小 | 不超过 `20MB` |
| 播放页风格 | `pixel / minimal / poster` |
| 播放方式 | 打开网页后展示播放器，用户点击播放 |
| 删除规则 | 删除文件后，文件与关联分享真实删除，所有相关旧链接失效 |

### 2.3 暂不支持
| 能力 | 说明 |
|------|------|
| 自动播放保证 | 移动浏览器通常限制自动播放 |
| 防下载 | 无法真正防止下载，只能避免默认打开即下载 |
| 视频转码 | 首版不做压缩、转码、多清晰度 |
| 视频封面 | 首版不做封面生成和封面展示 |
| 批量操作 | 首版不做批量上传、批量写卡、批量删除 |

## 三、用户流程
### 3.1 小程序端：上传并写入
```text
用户进入 NFC 首页
    ->
点击“本地音视频”
    ->
从微信会话选择本地音频或视频文件
    ->
系统校验格式和大小
    ->
用户可输入播放页标题（可跳过）
    ->
用户选择播放页风格
    ->
系统上传文件并生成播放页链接
    ->
打开 NFC 写卡弹窗
    ->
用户贴近 NFC 标签
    ->
写入成功
```

### 3.2 网页端：碰卡播放
```text
其他用户触碰 NFC 标签
    ->
浏览器打开 /play/{themeKey}/{shareId}
    ->
网页请求 /api/share/detail?id=shareId
    ->
云部署服务查询分享记录和文件记录
    ->
云部署服务生成当前媒体播放地址
    ->
网页根据风格和媒体类型渲染播放器
    ->
用户点击播放
```

### 3.3 小程序端：我的文件复用
```text
用户进入“我的文件”
    ->
查看已上传媒体文件
    ->
选择某个文件并点击“去写入”或“复制链接”
    ->
进入本地音视频写入页，或直接复制当前播放页链接
    ->
用户可重新输入标题或选择风格
    ->
系统生成新的播放页链接
    ->
写入 NFC
```

### 3.4 小程序端：删除
```text
用户在“我的文件”点击删除
    ->
系统展示确认提示
    ->
提示删除后，所有已写入 NFC 的相关链接都将失效
    ->
用户确认删除
    ->
系统真实删除文件和关联分享
    ->
旧 NFC 链接再次打开时提示“链接不存在或已删除”
```

## 四、功能需求
### 4.1 功能清单
| 编号 | 功能名称 | 需求描述 | 优先级 |
|------|----------|----------|--------|
| M01 | 本地音视频选择 | 用户可以从微信会话选择本地音频或视频文件 | P0 |
| M02 | 文件校验 | 系统拦截不支持格式、超大小和异常文件 | P0 |
| M03 | 媒体类型识别 | 系统识别当前文件是音频还是视频 | P0 |
| M04 | 标题输入 | 用户可选填播放页标题 | P0 |
| M05 | 风格选择 | 用户可以选择 `pixel / minimal / poster` 三种风格 | P0 |
| M06 | 链接生成 | 系统生成可写入 NFC 的播放页链接 | P0 |
| M07 | NFC 写入 | 系统将播放页链接写入 NFC 标签 | P0 |
| M08 | 我的文件 | 用户可以查看已上传媒体文件 | P0 |
| M09 | 再次写入 | 用户可以基于同一文件再次生成播放链接并写卡 | P0 |
| M10 | 复制链接 | 用户可以复制最近一次生成的播放页链接 | P0 |
| M11 | 删除文件 | 用户可以删除文件，删除后该文件关联的所有旧链接失效 | P0 |
| M12 | 浏览器播放页 | 浏览器打开链接后展示播放器页面 | P0 |
| M13 | 删除后提示 | 旧链接打开后显示“链接不存在或已删除” | P0 |

### 4.2 标题规则
- 标题为可选项
- 填写标题时，用于浏览器播放页主标题
- 标题建议限制为 `1-30` 个中文字符或等价长度
- 页面可以默认预填文件名，用户允许修改，也可以清空后直接生成链接
- 已填写标题时，播放页需要支持长标题自动换行
- 未填写标题时，网页端不显示标题区块

### 4.3 文件选择边界
- 首版通过微信能力从微信会话选择音视频文件
- 不承诺支持从系统文件管理器或手机相册任意选择本地文件
- 如果用户需要上传手机本地文件，需要先将文件发送到微信会话后再选择

### 4.4 风格规则
| 风格 | 页面方向 |
|------|----------|
| `pixel` | 像素风，默认风格，与小程序视觉一致 |
| `minimal` | 极简风，适合说明、名片、语音留言 |
| `poster` | 海报风，适合祝福、纪念、活动展示 |

- 风格是网页播放页的页面设计方向，不按音频或视频类型区分
- `pixel / minimal / poster` 三种风格都需要同时覆盖音频播放页和视频播放页
- 每种风格使用独立播放页目录和独立视觉稿，避免实现阶段混淆

风格与媒体类型矩阵：

| 风格 | 音频播放页 | 视频播放页 |
|------|------------|------------|
| `pixel` | 支持 | 支持 |
| `minimal` | 支持 | 支持 |
| `poster` | 支持 | 支持 |

### 4.5 播放页规则
- 播放页链接长期稳定
- NFC 不写入媒体文件直链
- 播放页始终保留媒体区和播放区；填写标题时才显示标题区
- 播放页 UI 不展示浏览器 URL 输入区
- 播放页静态资源与云部署服务一起提供，使用原生 HTML / CSS / JS
- 网页端通过云部署后端读取分享数据，由后端负责查询记录和生成媒体播放地址
- 音频页提供播放、暂停
- 视频页提供播放、全屏或重播
- 音频播放页与视频播放页分开展示设计稿
- 播放页正式链接固定为 `/play/{themeKey}/{shareId}`
- 删除后的旧链接仍打开播放页，但页面提示“链接不存在或已删除”

## 五、页面需求
### 5.1 小程序页面
| 页面 | 需求说明 |
|------|----------|
| NFC 首页 | 保留原有入口，新增或升级“本地音视频”入口 |
| 本地音视频写入页 | 完成选文件、选填标题、选风格、生成链接、写卡 |
| 我的文件 | 展示文件列表，支持去写入、复制链接和删除 |
| 写卡弹窗 | 复用现有 NFC 写入反馈 |

删除确认要求：
- 删除前必须二次确认
- 文案需明确：删除后，所有已写入 NFC 的相关链接都会失效
- 删除完成后，旧链接打开时提示“链接不存在或已删除”

### 5.2 网页播放页
说明：播放页运行在浏览器环境，不属于小程序页面栈。

| 场景 | 页面表现 |
|------|----------|
| 音频播放页 | 按 `pixel / minimal / poster` 三种风格分别提供音频页面设计 |
| 视频播放页 | 按 `pixel / minimal / poster` 三种风格分别提供视频页面设计 |
| 加载中 | 独立状态页，展示加载提示 |
| 链接不存在或已删除 | 独立状态页，展示错误提示 |
| 加载失败 | 独立状态页，展示重试入口 |

网页端设计要求：
- 网页端播放页需要与小程序端页面分开展示
- 音频播放页与视频播放页需要分开展示
- 每张播放页固定包含媒体区域和播放区域；填写标题时再显示标题区域
- 浏览器地址输入区域不属于网页 UI，不纳入设计图
- 网页端页面按 `pixel / minimal / poster` 分目录维护
- 网页端通过云部署服务同源提供静态页面和详情接口

## 七、版本关系
当前开发版中的 V2.0 本地音频方案尚未发布，不需要处理线上老数据。  
V3.0 直接替换未发布的 V2.0 本地音频方案，新上传内容统一使用播放页链接。
